# Performance Verification — 2026-07-14 (Worker W3.3)

Scope: verify the June audit's perf findings against the LIVE site at
https://mei-hagalil-gis.vercel.app **after** Waves 1+2 (commit `93f8fcb`,
deployed — confirmed byte-identical to the local working tree at HEAD via a
live `diff`, see Methodology). Read-only: no code changes, no commits. Live
probes were anonymous GETs/HEADs, 39 total (budget: ~40).

---

## Methodology & limits (read this before the numbers)

- **Live vs local**: fetched the live `index.html` and diffed it byte-for-byte
  against the repo's `index.html` at HEAD — **identical** (25,044 bytes both
  sides; live `Last-Modified` matches the commit timestamp). This means local
  file sizes (`wc -c` / `git cat-file -s`) are a reliable stand-in for live
  raw bytes, so the request budget was spent on **compression ratios and
  timings**, not re-downloading every one of the ~60 JS/CSS files individually.
- **Compressed-size estimate for local assets**: sampled 6 eager JS files, 3
  lazy JS files, and all 3 local CSS files live with `Accept-Encoding: gzip,
  br` (Vercel served Brotli in every case), computed each file's
  compressed/raw ratio, averaged per bucket (eager JS ≈0.354, lazy JS ≈0.302,
  CSS ≈0.226), and applied that average to the files not individually
  fetched. This is a **single-sample-per-bucket estimate**, not a full census
  — real per-file ratios vary with code repetition/JSON-literal density. As a
  sanity check, applying this method to the **pre-Wave1+2** codebase
  reproduces the June audit's own headline figures almost exactly (see
  §1) — that agreement is the main evidence the estimate is sound.
- **CDN assets**: measured directly (HEAD/GET), so those numbers are exact
  wire bytes, not estimates.
- **No headless browser / no Lighthouse.** "First paint" in §2 is a
  network-timing approximation from curl (DNS/connect/TLS/TTFB), run
  serially, from this sandbox's network location — not a real end-user
  Core Web Vital, and not from a Galilee 4G client (the audit's original
  basis for "FCP 0.8–1.5s"). Treat §2 as bounding the CSS-fetch slice of that
  budget, not replacing the audit's estimate.
- `curl`'s local libcurl build has **no HTTP/2 support** (`--http2` errors
  out before connecting), so all timings are serial, single-connection
  HTTP/1.1 measurements. A real browser negotiates HTTP/2 (or 3) with Vercel
  and multiplexes same-origin requests, so real same-origin fetch times are
  almost certainly better than the serial numbers below suggest — flagged
  wherever it matters.

### 1b. Before vs after Waves 1+2 (the audit's own basis, local-only)

The June audit's claim — **"30+ separate local JS files… ~500 KB
uncompressed; ~150–200 KB gzipped"** — was scoped to first-party (local)
eager JS+CSS, not CDN libraries. Re-deriving that exact same slice at the
pre-Wave1+2 commit (`93f8fcb^`) and at today's live commit:

| | Files | Raw | Compressed (est.) |
|---|---:|---:|---:|
| **Before** (pre-Wave1+2) | 33 JS + 3 CSS = 36 | 539,562 B (527 KB) | ~180,169 B (176 KB) |
| **After** (live today) | 36 JS + 3 CSS = 39 | 627,224 B (613 KB) | ~211,202 B (206 KB) |
| **Δ** | **+3 requests** | **+87,662 B (+16.3%)** | **+31,033 B (+17.2%)** |

The "before" row lands almost exactly inside the audit's own "~500 KB /
150–200 KB" range (527 KB / 176 KB), which is the cross-check that the
compression-ratio methodology in §Methodology is trustworthy. On that same
basis, **Waves 1+2 pushed the eager local payload past the audit's own
upper gzip bound** (206 KB > 200 KB ceiling) — a real, quantifiable
regression, not just added functionality:

- **+3 eager requests**, all new first-party files promoted to the
  critical path: `js/crs-utils.js` (4.8 KB), `js/layer-naming.js` (2.0 KB),
  `js/gis-realtime.js` (18.3 KB — the new attribute-table realtime module).
- Remaining growth (+87,662 − 25,117 for the 3 new files = **+62,545 B**) is
  organic growth inside existing eager files — `gis-feature-table.js`
  (+528/−37 lines), `gis-engine-sidebar.js` (+223/−11), `gis-edit.js`
  (+196/−8) are the biggest contributors per `git diff --numstat`.
- **Lazy JS also grew**: 265,279 → 301,939 bytes raw (+36,660, +13.8%),
  driven mostly by `export-feature.js` (+481/−164 lines) and
  `export-formats.js` (+199/−9) — both lazy, so this growth doesn't touch
  first paint, only the idle-load bundle.
- CSS and `index.html`'s own weight are essentially flat (CSS: 0 files
  touched by Waves 1+2 at all; HTML: +682 B for the 3 new `<script>` tags).

**Honest framing**: this is a genuine ~17% regression against the audit's
own gzip ceiling, but it bought real architectural fixes — server-side
attribute-table paging and scoped realtime channels (§3) — that *reduce*
runtime data transfer far more than 31 KB of extra script weight costs on
load. The trade was worth it; it should still be tracked so it doesn't
compound silently over future waves (see §5).

## 2. Render-blocking CSS analysis

7 stylesheets block first paint — unchanged in **count** since before
Waves 1+2 (Waves 1+2 touched zero CSS files; the audit's "6" was already
slightly stale even in June, since `arcgis-pro.css` was already present
pre-wave):

| File | Origin | Wire bytes (br/gzip) |
|---|---|---:|
| `leaflet.min.css` | cdnjs.cloudflare.com | 2,249 |
| `MarkerCluster.css` | cdnjs.cloudflare.com | 210 |
| `MarkerCluster.Default.css` | cdnjs.cloudflare.com | 310 |
| `leaflet-geoman.css` | unpkg.com | 6,393 |
| `css/shared.css` | own origin | 5,208 |
| `css/pages/index.css` | own origin | 6,126 |
| `css/pages/arcgis-pro.css` | own origin | 7,839 |

These span **3 distinct origins** (own + cdnjs.cloudflare.com + unpkg.com),
each requiring its own DNS+TCP+TLS handshake before that origin's CSS can
land, gating CSSOM construction (all `<link rel=stylesheet>` tags are
synchronous, no `media`/`preload` tricks). Measured cold-connection timings:

| Origin | dns | connect | tls (cumulative) | ttfb (cumulative) |
|---|---:|---:|---:|---:|
| mei-hagalil-gis.vercel.app | 0.020–0.026s | 0.027–0.033s | 0.15–0.17s | 0.23–0.25s |
| cdnjs.cloudflare.com | 0.041s | 0.112s | 0.209s | 0.308s |
| unpkg.com | 0.018s | 0.085s | 0.179s | 0.274s |

In a real browser these three origin-chains resolve **in parallel**, not
serially — first paint is gated by the *slowest* chain, not the sum. From
this vantage point that's roughly the cdnjs chain (~0.31s to first byte),
plus the HTML TTFB (~0.23–0.25s) needed before the parser even discovers the
`<link>` tags. Call it a **~0.5–0.6s network-only floor for CSSOM-ready**
from here — this is *not* a real FCP (no paint timer, no real client
device/network, wrong geography for the audit's 4G-Galilee basis) and should
not be quoted as one. What it does show cleanly: **the CSS critical path is
untouched by Waves 1+2** (0 CSS files changed) and remains bounded by a
3-origin fan-out that vendoring the CDN CSS locally would collapse to 1
origin — a low-effort, bundler-independent fix (see §5).

Additional wrinkle not in the original audit: `vercel.json` sets
`Cache-Control: no-cache` on **every** local JS/CSS/HTML response (confirmed
live on all sampled requests). Combined with `sw.js`'s
stale-while-revalidate strategy for same-origin static assets, this means
first-time/non-PWA visits pay a full revalidation round-trip per local file
on every load, while repeat visits *with an installed/active service worker*
get a near-instant cache hit — the audit's own "SW vs Vercel caching
tension" note is confirmed still present and unaddressed by Waves 1+2.

## 3. Fan-out & fetch-shape wins — code-level, before vs after

### 3a. Incidents realtime (`js/pages/index.js`)

- **Before**: `subscribeRT()` called `renderAll()` synchronously inside the
  `postgres_changes` callback — **every single event → one render**, 1:1,
  no coalescing. A burst of N events (e.g. a bulk status change) meant N
  full re-renders of the map markers + both list panels + stat counters.
- **After**: the callback (`_onIncidentRT`) now calls
  `_renderAllDebounced()`, a 300ms trailing debounce
  (`js/pages/index.js:865-868`). Any burst of events landing within a
  rolling 300ms window collapses to **exactly 1 render** — e.g. 20 events
  within 300ms → 1 render instead of 20 (20× fewer renders for that burst).
  Isolated events spaced further apart than 300ms see no change (still 1:1)
  — this is a burst-shaped win, not a universal one.
- **Village-scoping is confirmed inert today.** `_incidentsChannelVillages(profile)`
  only returns a per-village filter list when `profile.role==='viewer'` AND
  `profile.assigned_villages` is a non-empty array
  (`js/pages/index.js:846-852`). A repo-wide grep for `assigned_villages`
  finds it **nowhere** in `gis-engine/sql/schema.sql` or any migration —
  the `profiles` table has no such column yet. So for 100% of users today,
  this function returns `null` and the code falls back to the exact same
  single unfiltered `inc-rt` channel that existed before Waves 1+2. The
  code's own comments correctly call this a "forward-compatible no-op" — it
  needs a follow-up migration (`profiles.assigned_villages`) before it does
  anything. This matches the task's expectation exactly: it is real,
  tested code (`test/security/incidents-rt.test.js`), but zero live impact
  until that column exists.

### 3b. Attribute table (`js/gis-feature-table.js` + new `features_page` RPC)

- **Before**: `GIS.features.getFeatures(layerId, 1000000)` →
  `features_geojson` RPC with `p_limit=1000000` — a **full-layer dump** in
  one response, capped only by a million-row ceiling that's effectively "all
  of it" for any real layer, regardless of how many rows the UI ever
  displayed.
- **After**: `features_page` RPC, **hard-clamped server-side to ≤500 rows**
  (`v_limit := LEAST(GREATEST(COALESCE(p_limit,500),1),500)` in
  `gis-engine/sql/migrations/2026-07-14-feature-table-pagination.sql:148`),
  plus a separate cheap `features_page_count` call for the total.
- **Bytes/row estimate**, grounded in the actual RPC response shape
  (`type:'Feature'`, `ST_AsGeoJSON` geometry, `properties` merged with
  `asset_code`/`__id`/`__layer_id`/`__edited_by`/`__edited_at`) — built two
  representative synthetic rows matching that exact shape and measured them:
  a Point feature (5 properties) serializes to **422 bytes**, a 4-vertex
  LineString (6 properties) to **516 bytes**. Call it **~470 bytes/row**
  average for typical water-network features.
- **For a 20,000-feature layer**: before = 20,000 × 470B ≈ **9.4 MB** of
  JSON transferred (and `JSON.parse()`'d on the main thread) just to *open*
  the table — this is exactly the tab-freeze risk the Wave 1+2 changelog
  calls out. After = 500 × 470B ≈ **235 KB** for page 1 — a **~40×
  reduction (97.5% fewer bytes)** for the same table-open action,
  independent of how large the underlying layer actually is.

### 3c. Realtime channels — old vs new vs the incidents anti-pattern

| | Channels for GIS `features` data | Filtering | Lifecycle |
|---|---|---|---|
| **Before Waves 1+2** | 0 | n/a | n/a — edits from other users were invisible until manual refresh |
| **After (`js/gis-realtime.js`)** | ≤2 concurrent (LRU-evicted) | server-side `layer_id=eq.<id>` per channel | only while that layer's attribute table is open (ref-counted watch/unwatch) |
| **Incidents (`inc-rt`, unrelated table)** | 1 | none — every row, every client | always on for every authenticated session, regardless of whether the incidents panel is visible |

The new `features` channels are a materially tighter design than the
pre-existing incidents pattern within the very same wave: filtered
server-side (not just client-side discard), capped at 2 concurrent, and
scoped to an actual UI lifecycle rather than "on for the whole session."
They also add own-write echo suppression (3s window, authorship-aware —
never suppresses a genuinely different author even inside the window) and a
2s trailing debounce per layer (same shape as §3a's 300ms one, coarser
window). The incidents channel remains the one broad always-on subscription
in the app — it's just for a small, bounded table (open incidents), so its
fan-out cost is inherently much lower than an unscoped `features` channel
would have been.

## 5. Verdict

### Top-3 remaining opportunities (ranked by impact/effort)

1. **Fix `Cache-Control` on static assets (HIGH impact / LOW-MEDIUM effort).**
   Every local JS/CSS/HTML response is `no-cache` today (§2), forcing a
   revalidation round-trip on **every** load of **every** one of the 39
   eager first-party files, for every visit without an active service
   worker. This is a bigger lever than raw byte count for an internal tool
   used daily by the same field crews (repeat visits dominate real usage).
   Fix doesn't need a bundler: keep the existing `?v=N` cache-busting
   convention but set `Cache-Control: public, max-age=31536000, immutable`
   on the versioned URLs, `no-cache` only on the unversioned HTML entry
   point — the same pattern already used correctly for the CDN libraries
   (§1a shows those already carry `max-age=31536000, immutable`).

2. **Vendor the 4 CDN CSS files locally (MEDIUM impact / LOW effort).**
   Collapses the 3-origin CSS fan-out in §2 to 1 origin, removing 2
   DNS+TCP+TLS handshakes from the render-blocking chain. The project
   already vendors one dependency this way (`js/vendor/togeojson.js`), so
   there's precedent and no new tooling required.

3. **Bundle/minify via esbuild (MEDIUM impact / LOW-MEDIUM effort, ~1 day
   per the original audit estimate).** §1b shows eager local JS+CSS has
   grown to ~613 KB raw / ~206 KB gzipped across 39 requests — already past
   the audit's own 150–200 KB gzip ceiling. A build step would plausibly
   cut both (no minification, no dead-code elimination, no tree-shaking
   observed in any sampled file today) and collapse the request count from
   39 to a handful.

### Build-step recommendation: **stay zero-build for now**

Do #1 and #2 first — both are bundler-independent, low-effort, and address
what this session's numbers show actually hurts more directly than raw byte
count: repeat-visit round-trips (no-cache headers) and first-paint origin
fan-out (3-origin CSS). A bundler alone wouldn't fix either.

The zero-build regression is real but not (yet) severe: +17% eager gzip
bytes and +3 eager requests from Waves 1+2, landing just past the audit's
own ceiling rather than blowing through it. Given HTTP/1.1→2/3 multiplexing
on Vercel (unverified locally — no HTTP/2 in this sandbox's curl — but
near-certain in any real browser) already blunts the classic "30+ requests"
overhead concern that motivated bundling in the 2015-era performance
playbook, the request-count argument for a bundler is weaker today than the
byte-count one.

**Trigger to revisit**: if the eager local gzip payload keeps growing at the
Wave 1+2 rate (+17%/wave), two more comparable waves would push it past
~280 KB gzip — meaningfully beyond the audit's original ceiling and into
territory where a build step's ~50–60% byte cut (per the audit's own
estimate) stops being optional. Track `eager local JS+CSS raw bytes` and
`eager request count` as the two numbers that would justify that call;
today's is 627,224 B / 39 requests.
